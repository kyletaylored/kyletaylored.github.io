I've always wanted a rover. Not the Mars kind — the kind you build yourself, flash firmware onto at 2am, and watch drag itself across a desk because you got the motor polarity wrong.

This is a build log for my ESP32-based Wi-Fi rover. Full camera stream, real-time control via a web UI, OTA firmware updates, and enough duct tape in the architecture to keep it running indefinitely.

Wiring it up

The ESP32-CAM doesn't have many free GPIO pins after the camera takes its share. I ended up using GPIO12, 13, 14, and 15 for motor direction control — which caused exactly the boot-loop issue you'd expect if you didn't pull GPIO12 low at startup.

rover.ino 
// Motor pins — GPIO12 needs pull-down at boot
#define MOTOR_A1 12
#define MOTOR_A2 13
#define MOTOR_B1 14
#define MOTOR_B2 15

void forward() {
  digitalWrite(MOTOR_A1, HIGH); digitalWrite(MOTOR_A2, LOW);
  digitalWrite(MOTOR_B1, HIGH); digitalWrite(MOTOR_B2, LOW);
}

The web UI

The control interface is served directly from the ESP32 — a single-page HTML file stored in SPIFFS. It uses a gamepad-style layout with arrow buttons and a live MJPEG stream in the corner. Touch events work on mobile, which was harder than it sounds.

Lessons learned: power your motors separately from your microcontroller. Ground loops are real. A $0.10 capacitor across each motor terminal can save you hours of debugging phantom resets.

Lesson learned

A 100µF capacitor across each motor terminal eliminates most EMI-induced resets. Do this before debugging anything else.
